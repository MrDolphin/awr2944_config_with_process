# TI AWR2944 毫米波雷达项目性能优化全纪实

> **项目背景**：基于 TI AWR2944P 的毫米波雷达配置与实时处理系统，原始版本在 200ms (5Hz) 下存在严重数据延迟、目标跟踪极易丢失以及多普勒计算精度不足等问题。

---

## 🛠️ 第一阶段：核心算法与物理公式修正 (Analysis & Tuning)

### 1. 测速上限瓶颈修复 (Bug Fix)
*   **定位问题**：原始代码在计算 `maxVelocity` (最大不模糊速度) 时，错误地除以了 `chirpsPerFrame`。
*   **影响**：导致雷达识别的运动上限仅为 `0.08m/s`，超过此速度目标即刻丢失。
*   **修复**：校准公式为 `maxVelocity = 3e8 / (4 * startFreq * 1e9 * chirpDuration)`。
*   **结果**：量程恢复至真实的 **~9.74m/s**。

### 2. 刷新率飞跃：从 5Hz 到 20Hz (Performance)
*   **改动**：将 MATLAB 内部定时器周期 `Period` 从 `200ms` 直接下调至 **`50ms`**。
*   **效果**：彻底消除了 UDP 数据包在后台排队产生的“视野时间差”，让可视化变得即时、顺滑。

### 3. 目标跟踪稳定性增强 (Robustness)
*   **低反射建轨**：`pointsThre` (初始化点数门限) 由 `5` 降至 **`2`**，解决了侧身时因点数不足导致的“视而不见”。
*   **长久记忆力**：`act2freeThre` (轨迹丢失门限) 由 `3` 猛增至 **`20`** (约 1 秒容错)，解决了目标快速移动或短暂遮挡导致的轨迹闪烁。

## 🚣‍♀️ 第四阶段：从室内模型到外场实战的架构大重构 (v2.0 Architecture Transition)
> **日期: 2026-03-23**
> **核心目标**: 彻底排查前端轨迹卡死飘移问题，抛弃不可靠的硬件追踪，自研软件解算层，并打通底层离线数据重演链路。

### 1. 轨迹真理之战：前端全自研 DBSCAN (Tracker Rewrite)
*   **问题**：早期版本强依赖雷达硬件吐出的 Type 7 (Target List)，但 AWR2944 该固件下输出的其实是驻留区域中心（坐标长期为 0），导致海量幽灵目标重叠。
*   **重构**：在 `radar_app.html` 中彻底切断硬件追踪，白手起家手写了一套 **JavaScript DBSCAN 聚类与连续多目标追踪器 (TargetTracker)**，实现了每一个目标的平滑追踪和独立颜色连贯航线绘制。
*   **盲点与修复**：为了消除室内极其恶劣的墙壁“多径镜像”，曾一度将聚类半径拉大到 `3.5m`，结果导致真实人体和镜像被平均计算，航迹死死卡在房间中央。现已精准调回正常船体尺度的 **`1.2m`**，配合 `> 0.05m/s` 的微小多普勒过滤，完美解封了大范围画圆轨迹，并在空房间内实现零误报。

### 2. 后端基建：彻底终结“串口僵尸死锁” (Backend Deadlock Fix)
*   **问题**：每次按下 Ctrl+C 强杀程序并重新运行时，`radar_server.py` 都会莫名其妙长睡不醒（假死），必须退出再重进才能苟活。
*   **根源**：强杀时雷达仍在以 3Mbps 速率向 Linux 内核灌数据，重开后 Python 读到了截断的旧包遗骸，将乱码解析成了动辄几个 GB 的包长，从而陷入了“无限等待数据补齐”的跨世纪单相思死锁。
*   **神级补丁**：
    1. 连上串口第一秒，立刻执行 `data_port.reset_input_buffer()` 核弹级清空旧账。
    2. 加入 `total_packet_len < 40 or > 1000000` 长度变态锁，一旦触发直接手起刀落切掉废弃的魔法字强迫从头寻找，脚本自此如丝般顺滑，再无卡顿。

### 3. 终极分析兵器：底层二进制解析与录制全家桶 (Offline Data Pipeline)
*   **痛点**：网页前端的数据为了性能丢弃了多维特征，无法用于算法二次验证；而原厂的 `load_radar_data.m` 只能读截获卡的 ADC，读不了串口的点云。
*   **双管齐下**：
    1. **Python 落盘引擎**：为 `radar_server.py` 加上了 `--log_file` 参数。若是 `.jsonl` 则无损保留全浮点 $(X,Y,Z,V)$，若是 `.bin` 则原封不动地暴写 UART Binary 原生字节流。
    2. **MATLAB 解包脚本**：我在本地专为您量身定做了 `parse_uart_bin.m` 脚本！它能在 MATLAB 中秒级完成魔法字同步、解帧、以及把点云的坐标和速度重组为了结构体，甚至还带上了一个绝美的动态 3D 播放器彩蛋，彻底打通了算法重演的任督二脉。

### 4. 噪音清洗：CFAR 阈值与 Debug 矩阵 (Hardware DSP Tuning)
*   **问题**：没人的时候空房间里竟然有 3 个稳如泰山的幽灵目标。这是因为之前的速度阈值设为 0，导致反光极强的死物被录入。
*   **硬件制裁**：修改了 `profile-2944.cfg`：
    1. 把 `cfarCfg` 阈值系数一举从 `15` 猛抬到 `18`，从纯物理层暴力砍断了室内多径散射中 90% 的虚弱回波。
    2. 开启了 `guiMonitor -1 1 1 0 0 1 1`，现已将一维距离图 (Range Profile) 和顶级二维距离多普勒热力图 (Range-Doppler Heatmap) 源源不断地压入 `.bin` 存档，为离线算法排雷提供了降维打击的分析材料。

## 🦾 第五阶段：从 PC 依赖到树莓派 4B 独立载荷的工业级部署 (RPI Edge Deployment)
> **日期: 2026-03-25**
> **核心目标**: 实现雷达服务器的脱机独立运行，彻底解决 SD 卡易损及网络地址漂移问题，打造“插电即用”的无人船载荷。

### 1. 硬件级稳定性：USB 3.0 磁盘阵地 (Storage Migration)
*   **挑战**：传统的 SD 卡在频繁读写雷达 binary 原始数据时极易发生分区损坏，且 I/O 延迟会导致 WebSocket 推流卡顿。
*   **重构**：将整个 **Raspberry Pi OS Lite (64-bit)** 运行环境迁移至 **USB 3.0 固态闪存** 启动。
*   **成果**：系统启动速度提升 2 倍，彻底杜绝了文件系统崩溃，为长时间大吞吐量雷达录制提供了物理保障。

### 2. 网络基建：静态 IP 与免密管理 (Network Strategy)
*   **问题**：DHCP 动态分配导致每次重启都要猜 IP，手机终端难以连接。
*   **对策**：
    1. 使用 `NetworkManager (nmcli)` 硬编码 **静态 IP 172.20.10.10**，确保在 `WLAN-DHSYS` 办公 WiFi 下永久固定。
    2. 配置 **SSH ED25519 密钥对**。
*   **成果**：笔记本/手机即便在代理开启的情况下也能通过直连 IP 瞬时访问，实现了 AI Agent 与树莓派的自动化指令闭环。

### 3. “插拔即自启”：基于设备绑定的 systemd 逻辑 (Service Engineering)
*   **设计原则**：雷达不插，服务不跑；雷达一插，自动起飞。
*   **实现**：编写了专用的 `deploy/radar.service` 配置文件。
    - **逻辑内核**：使用 `BindsTo=dev-ttyACM0.device`。系统内核一旦识别到雷达板的 USB 序列号，Systemd 会在 3 秒初始化延迟后自动拉起 `radar_server.py`。
    - **环境隔离**：通过 `Environment` 路径映射，解决了 `pi` 用户本地 Python 依赖库的路径冲突。
*   **现状**：实现了真正的无人值守。哪怕树莓派先开机，后插雷达，后台解析引擎也会在探测到串口的第一时间秒级响应。

### 4. 仓库洁癖：.gitignore 与目录重构 (Repo Engineering)
*   **痛点**：25MB+ 的 `.bin` 录制文件常误入 Git 缓存，导致仓库体积爆炸。
*   **优化**：
    1. 新增 `.gitignore`：精准拦截 `record/*.bin` 和 `matlab_exe/*.asv`。
    2. 发布目录规范化：新增 `deploy/` 存放运维脚本，`matlab_exe/` 集中存放 GUI 备份，保持根目录清爽。

## 🌊 第六阶段：面向外场部署的通信链路强固与交互升级 (Field Deployment Hardening)
> **日期: 2026-04-15**
> **核心目标**: 面向码头、无人船和外场环境，进一步解决网络漂移、串口翻转、强光可视性不足和配置管理分散等实战问题，让系统从“可运行”走向“可部署、可维护、可演示”。

### 1. 外场部署可靠性固化：树莓派 4B 独立承载能力增强 (Deployment Reliability)
*   **问题**：外场部署对系统稳定性提出更高要求，传统 SD 卡易损、恢复慢，不利于高频日志写入和长期运行。
*   **优化**：
    1. 固化 **Raspberry Pi OS Lite (64-bit)** + **USB 3.0 存储启动** 的部署路线。
    2. 明确树莓派作为雷达服务独立载荷节点，承接采集、配置下发与 WebSocket 推流。
*   **结果**：部署链路更稳定，为长期录制、外场调试和无人值守运行打下基础。

### 2. 网络链路强固：静态地址、免密维护与快速重连 (Connectivity Hardening)
*   **问题**：真实外场中 DHCP 飘移、终端切网和断连后难以快速恢复，严重影响调试与演示体验。
*   **优化**：
    1. 为 `wlan0` 继续采用静态 IP `172.20.10.10`，避免重启后地址漂移。
    2. 配置 SSH 公钥免密登录，方便远程维护与自动化运维。
    3. 将前端默认 WebSocket 地址同步更新至固定 IP。
    4. 在前端引入 **LocalStorage 历史连接缓存机制**，保存已成功连接的节点地址，便于外场断连后的快速重连。
*   **结果**：设备连接路径更短，演示与维护过程中的网络不确定性显著下降。

### 3. 后端通信重构：串口自动发现与握手自适应 (Backend Communication Refactor)
*   **问题**：外场插拔和重新上电后，`/dev/ttyACM0` 与 `/dev/ttyACM1` 可能翻转，原先依赖手工指定串口会导致系统无法启动；同时配置命令响应时间差异大，容易引发握手阻塞。
*   **重构**：
    1. 在 `radar_server.py` 中引入 **`auto_detect_ports()` 自动探测逻辑**，通过发送空指令并监听 `mmwDemo:/>` 回显，自动识别配置口和数据口。
    2. 重构串口握手流程，移除不稳定的 DTR / RTS 硬件级流控依赖。
    3. 针对不同类型的雷达命令增加动态等待与超时容差，提升配置下发成功率。
    4. 对配置文件路径进行统一收口，集中归档至 `Config/` 目录，并在 WebSocket 接口中同步适配读取、保存和应用逻辑。
*   **结果**：系统对外设插拔、端口漂移和高负载命令下发的适应性显著增强，部署和重启过程更平滑。

### 4. 前端交互与可视化升级：外场强光下的可读性增强 (Frontend Visibility Upgrade)
*   **问题**：原始点云绘制样式在海面高亮、强反光或室外强光环境下可视性不足，难以快速辨识目标和轨迹。
*   **优化**：
    1. 重构点云渲染样式，采用更高亮度、更大尺寸的发光散点绘制方式。
    2. 保留并强化软件追踪器输出的目标航迹与关键信息叠加，提高强光场景下的观测性。
    3. 对脏数据目标包进行旁路处理，继续坚持由前端纯 JavaScript Tracker 统一生成稳定目标轨迹。
*   **结果**：前端界面更适合户外实时观测，目标、轨迹和状态信息更易辨认。

### 5. 配置体系扩展：为模式验证和远距离探测预留试验入口 (Configuration Expansion)
*   **目标**：为后续 4T4R、MIMO、beamforming 等模式评估建立配置基础。
*   **新增**：
    1. 引入 `beamforming_scan.cfg`，作为电子扫描和波束控制试验配置。
    2. 引入 `mimo_4tx_full.cfg`，作为四发四收高负载点云模式的预研骨架。
*   **现状**：配置资产已开始从“单一默认配置”向“多模式实验配置库”演进，但高负载模式仍需结合内存占用、`frameCfg` 和 SoC 资源限制继续调试。

### 6. 阶段成果总结
这一阶段的工作，核心不是单点优化，而是把系统向“真实外场可部署”推进了一步，具体体现在：

*   **树莓派部署更稳**：系统可作为独立载荷长期运行。
*   **网络连接更稳**：固定地址 + 历史连接缓存减少了外场断连损耗。
*   **串口链路更稳**：自动识别配置口和数据口，降低人工介入。
*   **前端观测更稳**：在强光和复杂背景下的可读性明显提升。
