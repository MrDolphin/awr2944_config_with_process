# TI AWR2944 毫米波雷达项目阶段汇报

## 一、项目定位

本项目围绕 TI AWR2944 毫米波雷达，构建一套面向船舶近距离感知的原型系统。  
当前技术路线已经从单纯算法验证，逐步转向“树莓派边缘部署 + 网页端实时显示 + 外场测试验证”的工程化原型。

结合近期外部交流反馈，项目后续更适合聚焦于：

**低成本毫米波辅助靠泊 / 自动靠泊 / 近距辅助感知**

相比通用海面远距探测，这一方向更贴近实际产品化需求，也更符合当前市场对于可靠性、成本和可落地性的要求。

---

## 二、当前已取得的阶段成果

### 1. 主链路已经打通

目前已经实现以下完整链路：

`AWR2944 雷达 -> 树莓派 -> WebSocket -> 网页端实时显示`

系统已具备：

- 雷达串口自动探测与配置下发
- 点云数据基础解析与转发
- Web 页面实时显示目标点云与轨迹
- 前端软件目标追踪能力

### 2. 树莓派部署能力明显增强

围绕真实外场部署，已完成一轮较完整的边缘部署优化：

- 系统迁移至 Raspberry Pi OS Lite 64-bit
- 采用 USB 3.0 存储启动，提升稳定性并降低 SD 卡损坏风险
- 固定树莓派静态 IP，降低网络漂移带来的连接中断
- 配置 SSH 免密维护，便于远程管理
- 通过 `systemd` 实现雷达设备插入即启动、拔出即停止

这意味着系统已经从“需要人工频繁干预”进入到“可独立运行、可重复部署”的阶段。

### 3. 通信链路与前端体验得到强化

针对外场常见问题，已做针对性优化：

- 增加串口自动识别机制，降低串口号翻转带来的启动失败风险
- 优化配置命令下发与握手超时控制，提高启动成功率
- 前端点云显示做了高亮和强光可视化优化
- 增加历史连接地址缓存，便于现场快速重连

### 4. 已具备外场推进基础

- 已完成外场环境初步踏勘
- `东海海事保障基地` 可作为后续测试候选场地
- 可优先在 `内海河道` 场景开展近距离感知与辅助靠泊验证

---

## 三、当前判断

项目目前已经完成“从 0 到 1”的关键验证，但还没有完全进入成熟产品阶段。

现阶段最重要的判断是：

1. 技术闭环已经初步成立，系统不是停留在概念上
2. 当前最有价值的方向，不是继续做大而全的海面监测，而是先聚焦近距辅助靠泊
3. 后续成败关键，不在于增加更多功能点，而在于提高可靠性、可复现性和外场表现

---

## 四、下一阶段建议

下一阶段建议以“辅助靠泊原型验证”为主线，按以下顺序推进：

### 1. 先做稳定演示版

- 完善前端配置管理、距离显示和连接切换
- 整理最小操作手册
- 形成一套可稳定演示的版本

### 2. 再做工程化重构

- 将当前单文件服务端逐步拆分成模块化结构
- 建立状态监控、日志、录制与回放能力
- 让系统从“能跑”变成“可维护”

### 3. 建立外场数据闭环

- 每次测试都同步记录配置、原始数据、截图和问题清单
- 支持离线回放与问题复盘
- 优先验证内海河道和靠泊近距场景

### 4. 最后做模式扩展

- 对 4T4R、MIMO、beamforming 等模式进行专项评估
- 以实际效果决定是否进入下一阶段深度投入

### 5. 同步推进算法优化

- 围绕辅助靠泊场景优化误报、漏报和轨迹连续性
- 建立内海河道模式、靠泊模式等场景化参数模板
- 通过录制回放不断验证参数调整效果

---

## 五、当前需要重点关注的三件事

### 1. 可靠性

系统已经具备原型能力，但外场长期运行的稳定性仍需继续验证，尤其是：

- 串口链路稳定性
- 切网和断连恢复能力
- 长时间运行时的异常自恢复

### 2. 算法有效性

下一步算法重点不是追求更复杂模型，而是先解决：

- 近距离低速目标是否稳定可见
- 靠泊场景下误报和漏报是否可控
- 轨迹是否连续、是否能用于真实辅助判断

### 3. 场景收敛

建议优先聚焦：

**码头 / 内海河道 / 低速近距 / 辅助靠泊**

这样更容易形成阶段成果，也更符合当前潜在合作方的兴趣点。

### 4. 成本与产品化路径

外部反馈表明，高成本远距雷达方案吸引力有限。  
因此更建议以“低成本毫米波辅助靠泊原型”为切入口，先验证需求、效果和可靠性，再决定后续扩展范围。

---

## 六、汇总结论

项目当前已经具备继续推进的基础，且方向上应从“通用海面探测”逐步收敛到“低成本毫米波辅助靠泊”。

简单概括就是：

**链路已通，原型已成，方向需要收敛，下一步重点是把系统做稳、把外场数据跑通、把近距靠泊场景验证出来。**
